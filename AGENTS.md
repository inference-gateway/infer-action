# AGENTS.md - Infer Action Project Guide

## Project Overview

This is a **composite GitHub Action** (no build step, no compiled runtime code) that runs the
[Infer CLI](https://github.com/inference-gateway/cli) agent against GitHub issues and comments.
The agent analyzes issues, makes code changes, creates pull requests, and posts progress updates
and results as comments using various AI providers.

**Key Technologies:**

- **GitHub Actions** - Composite action with bash steps glued together in `action.yml`
- **Infer CLI** - AI agent framework from Inference Gateway
- **Task** (`go-task`) - Task runner for development workflows
- **Flox** - Development environment management
- **Multiple AI providers** - Anthropic, OpenAI, Google, DeepSeek, Groq, Mistral, Cloudflare, Cohere, Ollama, Ollama Cloud
- **semantic-release** - Automated versioning and changelog generation
- **markdownlint** - Markdown linting

**License:** Apache-2.0

### Action Execution Flow (`action.yml`)

The action runs these steps in sequence, all gated on `steps.check-trigger.outputs.triggered == 'true'`:

| Step | Description |
| --- | --- |
| 1. **check-trigger** | Detects `trigger-phrase`; extracts `/model` override; skips bot comments. |
| 2. **Add eyes reaction** | Adds 👀 reaction to triggering comment (only on `issue_comment`). |
| 3. **cooking-message** | Deletes old cooking comments, posts fresh one; passes ID to agent. |
| 4. **install-cli** | Downloads and installs Infer CLI (version from `inputs.version`). |
| 5. **init-cli** | Runs `infer init --overwrite` to initialize the CLI. |
| 6. **configure-git** | Sets `github-actions[bot]` as commit identity. |
| 7. **run-agent** | **Load-bearing step.** Builds system prompt; runs `infer agent`. |
| 8. **Post results** | Posts final comment with status, model, exit code, output. |
| 9. **Cleanup** | Removes `/tmp/agent-output.txt` and `/tmp/failed-tool-calls.txt`. |

### Cross-Cutting Concerns

- **Bash whitelist**: The Infer CLI restricts shell commands via `INFER_TOOLS_BASH_WHITELIST_COMMANDS`
  and `INFER_TOOLS_BASH_WHITELIST_PATTERNS`. When `enable-git-operations: true` (default),
  `gh,git` and `^gh .*,^git .*` are prepended. Setting it to `false` strips git/gh entirely →
  comment-only mode. Anything the agent needs to run (npm, pytest, etc.) must be added through
  `bash-whitelist-commands` / `bash-whitelist-patterns` inputs.
- **Model override flow**: `check-trigger` outputs `model_override`; `run-agent` uses
  `steps.check-trigger.outputs.model_override || inputs.model`. The same fallback is repeated
  in the "Post results" step - keep them in sync if changing.
- **Branch logic in system prompt**: The embedded instructions tell the agent to check
  `git branch --show-current` and only create `fix/issue-{N}` + a PR when starting from
  `main`/`master`. Otherwise it commits to the current branch. This logic lives in the heredoc
  inside `action.yml`, not in any external file.
- **Heredoc quoting**: Issue title/body are interpolated via `cat <<'EOF'` (single-quoted
  delimiter) to prevent the issue content from being interpreted as shell. Preserve this when
  editing - unquoted heredocs are a command-injection vector.

## Testing Instructions

### Manual Testing Checklist

1. **Trigger Detection**: Verify the action correctly detects the trigger phrase (`@infer` by
   default) in both issue bodies and comments.
2. **Bot Author Skipping**: Confirm bot-authored comments are skipped (prevents recursion loops).
3. **Model Override**: Test `/model provider/name` syntax in issue bodies and comments.
4. **Cooking Message**: Verify the "I'm cooking..." GIF comment is posted and updated (not duplicated).
5. **Plan Posting**: Ensure the agent posts a plan with todos and updates it in real-time on the cooking comment.
6. **File Changes**: Verify the agent can read, edit, and write files appropriately.
7. **Git Operations**: Test branch creation, committing, and pushing.
8. **Pull Request Creation**: Verify PRs are created with correct title/body format when starting from `main`.
9. **Comment-Only Mode**: Test with `enable-git-operations: false` - agent should only post comments.
10. **Result Posting**: Verify final comment with status, model, exit code, and agent output.
11. **Error Handling**: Test various failure scenarios (invalid API keys, network errors, timeout).
12. **Custom Instructions**: Verify custom instructions are appended to the system prompt correctly.

### Test Event Fixtures

**`.github/workflows/events/issue-opened.json`** - Simulates an issue opened with `@infer` in the title.

**`.github/workflows/events/issue-comment.json`** - Simulates a comment containing `@infer` on an existing issue.

These fixtures are used by `task test:issue` and `task test:comment` respectively.

## Project Conventions and Coding Standards

### Conventional Commits (REQUIRED)

**All commit messages MUST follow the Conventional Commits format.** This is enforced by semantic-release for versioning.

```text
type(scope): Description

Common types: feat, fix, docs, style, refactor, test, chore
```

- `feat` → minor version bump
- `fix`, `docs`, `chore` → patch version bump
- `breaking:` → major version bump
- Scope describes the area of change (e.g., `api`, `ui`, `auth`, `config`)

Examples:

```text
feat(ci): Setup infer workflow
fix(workflow): Update system prompt to follow conventional commit format
docs(readme): Update model provider
chore(deps): Upgrade dev dependencies
```

### Bash Scripting

- Use POSIX-compliant shell scripting where possible
- Variables: `snake_case` or `UPPER_SNAKE` for environment variables
- Always use single-quoted heredoc delimiters (`<<'EOF'`) when interpolating user content to prevent command injection
- Prefer `[[ ]]` over `[ ]` for conditionals (bash-specific but available in GitHub Actions runners)

### GitHub Action Conventions

- Input names: **kebab-case** (e.g., `github-token`, `max-turns`, `trigger-phrase`)
- Output names: **kebab-case**
- Step IDs: **kebab-case** (e.g., `check-trigger`, `run-agent`)
- Use `composite` runs type
- Reference inputs as `${{ inputs.input-name }}`
- Reference step outputs as `${{ steps.step-id.outputs.output-name }}`

### YAML

- Use `---` document start marker
- Follow GitHub Actions best practices
- Use 2-space indentation

### Markdown

- Linted by `markdownlint` per `.markdownlint.json`
- Line length: 120 characters max
- `CHANGELOG.md` is excluded from linting
- Allows `div`, `p`, `br`, `img`, `a` HTML elements

**This is the entire product.** There is no compiled code - everything is bash steps in this

**Editing rules:**

## GitHub Actions Workflow: Infer

The main workflow (`.github/workflows/infer.yml`) is the dogfooding setup:

```yaml
name: Infer

on:
  issues:
    types: [opened, edited]
  issue_comment:
    types: [created]

permissions:
  issues: write
  contents: write
  pull-requests: write

concurrency:
  group: ${{ github.workflow }}-${{ github.event.issue.number }}
  cancel-in-progress: true
```

**Important:** This workflow uses a **GitHub App token** instead of `GITHUB_TOKEN`. This is
critical because PRs created by `GITHUB_TOKEN` don't trigger other workflows (like CI checks).
The GitHub App token ensures agent-created PRs trigger downstream workflows.

## Agent Instructions (System Prompt)

The agent receives a system prompt built dynamically in the `run-agent` step. Key instructions include:

1. **GitHub context**: Repository name, issue number, cooking comment ID
2. **Plan and progress tracking**: Create a plan with todos, update the cooking comment (not create new ones)
3. **Branch logic**: Check current branch - create `fix/issue-{N}` only if on `main`/`master`
4. **Conventional commits**: Must use conventional commit format for commits and PR titles
5. **PR creation**: Create PR only when starting from `main`/`master`
6. **Custom instructions**: Appended after the default instructions if provided

### How the Agent Should Behave

When an AI agent runs inside this action, it should:

1. Create a plan with checkboxes and post it by updating the cooking comment
2. Read and analyze the repository code
3. Make file changes as needed
4. Check the current branch:
   - On `main` → create `fix/issue-{N}`, commit, push, create PR
   - On feature branch → commit directly, no new PR
5. Update the cooking comment continuously with progress
6. Post final status

## Security Considerations

### Secrets Management

- **Never commit API keys** - `.env` is in `.gitignore`
- Use GitHub Secrets for all sensitive data in workflows
- The `.env.example` file contains placeholder values only

### Bash Whitelist

The Infer CLI restricts which shell commands the agent can execute. This is a security boundary:

- **Default whitelist includes safe commands** (ls, cat, git status, etc.)
- **Git operations** (gh, git) are only added when `enable-git-operations: true`
- **User-provided commands** via `bash-whitelist-commands` and `bash-whitelist-patterns` inputs
- Only whitelist commands you trust the AI agent to execute

### Heredoc Injection Prevention

Issue titles, bodies, and comment bodies are interpolated using **single-quoted heredoc
delimiters** (`<<'EOF'`). This prevents the content from being interpreted as shell commands.
**Never change these to unquoted heredocs** - doing so creates a command-injection
vulnerability.

### Input Validation

- Trigger phrases are compared with string matching
- Model override is extracted via regex
- Bot comments are detected by checking `user.login` for `[bot]` suffix and `user.type` for `Bot`

### Bot Detection

The action prevents recursion loops by skipping comments from bot users:

```bash
if [[ "$COMMENT_AUTHOR" == *"[bot]" ]] || [[ "${{ github.event.comment.user.type }}" == "Bot" ]]; then
  TRIGGERED=false
fi
```

## Release Process

Releases are **manual** via `workflow_dispatch` on the `release.yml` workflow:

1. Merge changes to `main` with conventional commit messages
2. Go to GitHub → Actions → Release workflow → "Run workflow"
3. `semantic-release` analyzes commits and determines the next version
4. `CHANGELOG.md` is updated and committed
5. A GitHub release is created with tag `v${version}`

**Version rules:**

- `feat` → minor (e.g., v0.3.0 → v0.4.0)
- `fix`, `docs`, `chore` → patch (e.g., v0.3.0 → v0.3.1)
- Breaking changes → major (e.g., v0.3.0 → v1.0.0)

### Adding a New AI Provider

1. Edit the `check-trigger` step in `action.yml`
2. Update the trigger phrase matching regex or logic
3. Update event type handling if needed
4. Test with `task test:issue` and `task test:comment`

### Enhancing the Agent System Prompt

1. Edit the `SYSTEM_PROMPT` heredoc in the `run-agent` step of `action.yml`
2. Test that the prompt is correctly constructed by checking agent behavior
3. Ensure heredoc quoting is preserved (single-quoted delimiters for user content)

### Changing Default Instructions

The default agent instructions are embedded in the `run-agent` step's heredoc. They cover:

- Plan creation with todos
- Real-time comment updates
- File change workflow (branch, commit, push)
- Pull request creation
- Result posting

Custom instructions from `inputs.custom-instructions` are **appended** to these defaults.

1. Make changes to `action.yml`
2. Run `task test:dry-run` to verify the workflow parses correctly
3. Run `task test:issue` or `task test:comment` to test locally

## Troubleshooting

### Common Issues

| Symptom | Likely Cause | Fix |
| --- | --- | --- |
| Action doesn't trigger | Wrong phrase or event type | Check `trigger-phrase` and `on:` events |
| Agent fails immediately | Missing/invalid API key | Verify correct key for chosen model |
| Comments not posting | Insufficient permissions | Ensure `issues: write` is set |
| PRs not created | Missing write perms | Add `contents: write` and `pull-requests: write` |
| Branch creation fails | No push access | Use GitHub App token instead of `GITHUB_TOKEN` |
| Agent creates new comments | Wrong comment ID | Verify `cooking_comment_id` is correct |
| `act` testing fails | Docker not running or wrong image | Use `catthehacker/ubuntu:act-24.04` |
| Command not found | Missing whitelist entries | Add commands via `bash-whitelist-commands` |

### Debugging

- Enable verbose act output: `--verbose` flag
- Check the "Post results" comment for agent output and failed tool calls
- The agent output is captured in `/tmp/agent-output.txt` during execution
- Failed tool calls are extracted with `jq` from the agent output JSON
