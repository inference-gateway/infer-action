# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This repo is a **GitHub composite Action** with a **TypeScript hot path**. The shipped surface is `action.yml` plus the pre-bundled ESM entrypoints under `dist/runner/`, `dist/recover/`, and `dist/post-results/` (committed to git, built from `src/*.ts` with `@vercel/ncc`). The "simple" steps (trigger detection, eyes reaction, cooking-comment post, CLI install, git config, cleanup) are still `shell: bash`. The three complex steps â€” running the agent with a parallel ticker that mirrors `TodoWrite` to the issue comment, recovering its work into a draft PR, and posting the final result â€” are `node` invocations of the bundled scripts.

Consumers reference it as `inference-gateway/infer-action@<ref>` from their workflows. When a GitHub issue or comment containing the trigger phrase (`@infer` by default) fires, the action installs the `infer` CLI, sets up Node 22, then runs `node dist/runner/index.js` (which spawns `infer agent` and observes its JSON-line stream, debouncing TodoWrite updates to the cooking comment), then `node dist/recover/index.js` on `always()` (which salvages any unpushed agent work into a draft PR â€” surviving a job-timeout cancellation that the runner can't â€” and links it), and finally `node dist/post-results/index.js` (which builds a structured result footer â€” including token usage and per-session cost â€” and PATCHes it into the same cooking comment).

## Common commands

The TypeScript hot path uses npm (no pnpm/yarn). `task` wraps the most common combinations.

```sh
npm ci                  # install (frozen lockfile)
npm run package         # ncc-bundle src/{runner,recover,post-results}.ts -> dist/
npm test                # vitest run
npm run typecheck       # tsc --noEmit
npm run lint            # eslint .
npm run lint:md         # markdownlint . (check-only; no --fix)
npm run format:check    # prettier --check .
npm run format:write    # prettier --write .
npm run all             # format + lint + test + typecheck + package
```

`task` targets:

- `task build` â€” `npm ci && npm run package`
- `task test:unit` â€” `npm test`
- `task test:mock SCENARIO=happy` â€” runs the runner end-to-end against the mock agent (`__tests__/fixtures/mock-agent.mjs`). Useful for local iteration without a real `infer` CLI or GitHub token. Mock scenarios: `happy`, `failures`, `no-todos`, `empty`.
- `task test:issue` / `task test:comment` / `task test:direct` / `task test:all` â€” `act`-based local tests that run the **working-tree** action (`uses: ./`) in `dry-run` mode against `examples/local/*.yml`. Require Docker + `act` only â€” no `.env`/token (dry-run simulates all mutations; reads fail-soft). Pass a token with `-s GITHUB_TOKEN=$(gh auth token)` to resolve real reads. `task test:list` lists jobs without executing; `task setup` checks `act`/Docker and seeds `.env`.
- `task lint` â€” `markdownlint --fix` (the auto-fixing local convenience; `npm run lint:md` is the same lint check-only, and is what CI runs. Both are separate from `npm run lint`, which is eslint over `src/`)
- `task clean` â€” removes `/tmp/agent-output.txt`

Run a single vitest file: `npx vitest run __tests__/failures.test.ts`
Run a single test by name: `npx vitest run -t "drops envelope failures with an empty message"`

CI (`.github/workflows/ci.yml`) runs format-check â†’ lint (eslint) â†’ lint:md (markdownlint) â†’ typecheck â†’ test â†’ package â†’ `git diff --exit-code dist/` â†’ mock smoke-test on every PR. The diff check fails if a contributor edited `src/` without rebuilding `dist/`.

Dry-run a build locally: set `dry-run: true` (and optionally `mock-agent-scenario: happy|failures|no-todos|empty`) on an action invocation. The action skips the CLI install/init/skills steps, points `INFER_BIN` at the bundled `__tests__/fixtures/mock-agent.mjs`, prints the resolved SYSTEM/TASK/REMINDER prompts, and **simulates** every GitHub mutation (`[dry-run] would â€¦`) while keeping reads real. Useful for eyeballing comment shape, the token-usage footer, and PR-link behavior of a build before cutting a release â€” without burning provider tokens or mutating anything. (`dry-run` is the only mock-agent path; the old `use-mock-agent` input was removed in favor of it.)

Releases run via the manually-dispatched `Release` workflow (`.github/workflows/release.yml`), which invokes `semantic-release`. Versions are derived from conventional commit messages per `.releaserc.yaml`.

## Architecture: what `action.yml` actually does

The composite action is one linear pipeline. The bash steps are unchanged from earlier versions; the three `node` steps (`run-agent`, `recover`, `post-results`) are where the TypeScript hot path lives.

1. **`check-trigger`** â€” scans `github.event.issue.{title,body}` or `github.event.comment.body` for the trigger phrase; skips if a bot authored the comment (recursion guard); extracts an optional `/model provider/name` override into `model_override`. Sets `triggered=true/false` â€” every later step is gated on this.
2. **Eyes reaction** + **cooking comment** â€” posts an "I'm cooking..." comment and captures its ID into `cooking_comment_id`. This is the single bot comment per run; everything downstream PATCHes it in place. Before posting, deletes any prior "I'm cooking" comments on the same issue (cleanup for runs that died mid-execution).
3. **Install + init Infer CLI** â€” `curl | bash` from the `inference-gateway/cli` repo at the pinned `version` input, then **Configure Infer for headless run** â€” `infer config set` to auto-approve the `Write`/`Edit`/`Delete` tools (v0.121.0 defaults them to require approval, which a headless run with no approval broker would otherwise block) and to set `tools.web_fetch.allowed_domains` from the `web-fetch-domains` input. This step also **hides `.infer/` from git** for the rest of the job (`.git/info/exclude` + `git update-index --skip-worktree`), so the config writes above don't dirty the working tree the agent commits from.
4. **Configure Git** as `github-actions[bot]`.
5. **Setup Node.js 22** via `actions/setup-node@v6` â€” required for the runner and post-results bundles.
6. **`run-agent`** (TS, `dist/runner/index.js`) â€” the core step. Invoked via `exec node â€¦` so the **node process is the step's top-level process** and actually receives `SIGINT`/`SIGTERM` when the job's `timeout-minutes` cancels it (a plain `node â€¦` leaves bash as top-level, and bash swallows the signal). Spawns `infer agent -m $INFER_AGENT_MODEL "$TASK"` (or `$INFER_BIN` if set â€” used by tests and by `dry-run: true`), tees stdout to `/tmp/agent-output.txt`, optionally mirrors it to the Actions log (off by default; `mirror-agent-logs`), and runs a **parallel ticker** over the JSON-line stream. The ticker dispatches to per-tool handlers â€” `TodoWrite` renders the agent's todos as a markdown checklist and **throttles** PATCHes to the cooking comment's plan zone (â‰¤ one per ~1.5 s, latest wins) and **persists the latest todos to `/tmp/infer-todos.json`** (the handoff the recover step reads) â€” plus, with `debug` on, a generic `Ticker.onMessage` listener that surfaces `compaction_started`/`compaction_completed` and reminder-injection events to the Actions log **regardless of stdout mirroring** (they flow on the muted stdout, so without this they stayed invisible even with debug on). On a normal exit the runner sets the `exit-code`/`run-duration-ms`/`result` outputs and returns. **It no longer recovers work or links PRs itself** â€” that moved to the `recover` step (7) so it survives a job-timeout cancellation. On `SIGINT`/`SIGTERM` (job timeout/cancel) the runner reaps the agent child, dumps a **breadcrumb** (the redacted tail of `/tmp/agent-output.txt`) to the log, and exits **without** setting `exit-code` â€” that emptiness is how the recover step detects a timeout. The action still appends the agent's PR-workflow writes (`git add/commit/push/checkout/switch/fetch` + `gh pr create`/`gh pr ready`) onto the CLI's read-only baseline via `INFER_TOOLS_BASH_ALLOW_APPEND` (the agent opens its own PR early as a draft on the happy path), deliberately withholding `gh pr merge`/`close`/`edit`/`review`.
7. **`recover`** (TS, `dist/recover/index.js`, runs on `always()`) â€” the model-independent **work-salvage + finalize** step (`recoverUnpushedWork`/`runRecovery` in `src/recovery.ts`). Because it is `always()` it runs on every outcome â€” happy exit, crash, or a job-`timeout-minutes` cancellation that killed `run-agent` mid-run; GitHub runs `always()` steps in the cancellation window (~4 min), so this survives a timeout that the **old in-`runner` recovery never did** (issue #58). If the agent left uncommitted/unpushed work and no PR exists, it commits, moves the work off `main` onto a `fix/issue-N` / `infer/auto-<run-id>` branch (or reuses the agent's branch / the PR head for `pull_request`), pushes, and opens a **draft** PR with a `buildPrBody`-synthesised body â€” then links it (`linkPr`) into the comment's middle zone and the `pr-url` output. On the happy path it instead finds the PR the agent already opened (`linkAgentPr`, with the thin-body backfill from `src/pr-body.ts`) and the recovery no-ops. It then emits the final `exit-code`/`run-duration-ms`/`result`/`stopped-early`/`timed-out` outputs that post-results and the action outputs consume (`finalizeStatus`): an **empty** `run-agent` exit-code â‡’ a timeout â‡’ `timed-out=true`, `stopped-early=true`, and exit-code **normalised to 0** (a recovered timeout is a soft âš ď¸Ź, not a âťŚ); the **stopped-early** signal (`detectStoppedEarly`) is otherwise any non-completed todo from the handoff file or â€” with git ops on â€” a still-dirty tree after recovery. Never merges; never pushes `main`/`master`; fork PRs are read-only and skipped. Its git/`gh` calls run with a 60 s timeout and `GIT_TERMINAL_PROMPT=0` so a hung push can't burn the cancellation budget.
8. **`post-results`** (TS, `dist/post-results/index.js`, runs on `always()`) â€” extracts failed tool calls from `/tmp/agent-output.txt` using a two-pass scan: pass 1 builds a `tool_call_id â†’ tool_name` map from every `assistant.tool_calls[]` message; pass 2 emits one row per failure, looking the name up via the map (so envelope failures get a real tool name instead of `(unknown tool)`) and **dropping any row whose error message is empty** (which fixed the "blank `(unknown tool):` rows" bug). Builds the status footer (icon, model, exit code, workflow link, the **agent's final response** â€” the `content` of the last assistant message with non-empty text, extracted by `extractFinalResponse` in `src/response.ts` and rendered as a visible, non-collapsed section directly under the status header and above the metadata, redacted and capped at 16,000 chars (omitted entirely when the run produced no closing text) â€” **token usage** summed from `token_usage` on assistant messages and **per-session cost** read from the CLI's session-end `session_stats` line â€” both via `extractUsage` in `src/usage.ts`; the parser admits that `type`-keyed line specifically, and cost renders only when non-zero â€” failures `<details>`), writes it to `$GITHUB_STEP_SUMMARY`, then PATCHes it into the cooking comment's result zone. Falls back to `POST` a new comment only if the PATCH fails or `cooking_comment_id` is empty. It reads the final status from the **recover** step (not run-agent, which sets no outputs when a timeout kills it). The status line is âś… Success, âťŚ Failed (non-zero exit), or âš ď¸Ź Stopped early â€” the last when the recover step's `stopped-early` signal (`INFER_STOPPED_EARLY`) is set on an exit-0 run; a **timed-out** run (`INFER_TIMED_OUT`, also from recover) is always âš ď¸Ź (never âťŚ, since exit-code was normalised to 0) with a "hit the job's time limit" note. âš ď¸Ź keeps the job's own conclusion unchanged.
9. **Cleanup** â€” removes the temp files (incl. `/tmp/infer-todos.json`).

Key cross-cutting concepts to keep coherent when editing:

- **Section sentinels in the cooking comment body** â€” `<!-- infer:plan-end -->` and `<!-- infer:result-start -->` split the body into three zones: plan (above plan-end), middle (between sentinels, used for the PR URL), result (below result-start, used for the final footer). Each writer (`runner.ts` ticker, `recover.ts` PR-link, `post-results.ts`) only ever replaces its own zone via `GithubClient.updateZone()`. This is what lets three independent writers update the same comment without stomping each other. See `splitZones` / `joinZones` in `src/github.ts`.
- **Ticker debounce semantics** are throttle-latest: first call sets a 1.5 s timer, subsequent calls update the pending value but don't reset the timer, the timer fires once with the latest value, and a new burst starts the next timer. Tuned so a tight loop of TodoWrites coalesces into â‰¤ one PATCH per ~1.5 s while still surfacing the freshest state. See `throttleLatest` in `src/ticker.ts`.
- **The bash allow-list is "CLI baseline + a git-write append."** As of CLI v0.121.0 the CLI owns the read-only baseline (`tools.bash.mode.all.allow`: file reads, read-only git/gh, and `gh project list|view|item-list|field-list`), inherited by headless `standard` mode â€” the action no longer ships its own read-only defaults. It only composes a small append in `src/bash-allow.ts` (`composeBashAllowAppend` / `GIT_WRITE_ALLOW`): the PR-workflow writes (`git add/commit/push/checkout/switch/fetch` + `gh pr create`/`gh pr ready`) when git ops are on, plus the `bash-allow-append` input, passed as the CLI's single append env var `INFER_TOOLS_BASH_ALLOW_APPEND`. `gh pr merge`/`close`/`edit`/`review` are never appended, so the agent can open and ready a draft PR but never merge it. Each entry is a Go regex anchored to the whole command (so `npm` matches only bare `npm`; write `npm( .*)?`); the CLI splits the list on `,` and `\n`. There is **no env-var path to replace the baseline** â€” to do that, configure the CLI's `.infer/config.yaml` directly.
- **`enable-git-operations: false`** disables recovery/PR-linking in the recover step *and* omits `GIT_WRITE_ALLOW` from the append, so the agent keeps only the CLI's read-only baseline (`bash-allow-append` still applies) and cannot push or open PRs by hand.
- **The runner owns the comment; the agent owns the branch and the PR on the happy path â€” with the `recover` step as a model-independent safety net.** The system prompt does not ask the agent to comment on the issue (the runner mirrors TodoWrite and posts the result), but it *does* tell the agent to commit on a `fix/issue-N` branch, run the repo's checks before each commit, and open the PR itself **early as a draft** with `gh pr create --draft --body-file` (writing the body to a file with the Write tool first, to avoid shell-quoting problems) and a real description, marking it ready with `gh pr ready` only when finished â€” never merging it. Opening early means an interrupted run still leaves a **draft** PR with the pushed work (rather than nothing); the recover step's read-only `stopped-early` check then surfaces a âš ď¸Ź status. **But weaker models skip this whole flow**, so the **recover step**'s `recoverUnpushedWork` (`src/recovery.ts`, see step 7) is the backstop: it commits/pushes whatever the agent left and opens the draft PR itself, never pushing `main` and never merging. If a weak model instead leaves a thin body, the recover step backfills one from the commit log (`src/pr-body.ts`). `custom-instructions` is still appended verbatim.
- **The `infer` binary path is overridable via `INFER_BIN`** â€” the integration test workflow and the local `task test:mock` target use this to substitute `__tests__/fixtures/mock-agent.mjs` for the real CLI.
- **Trigger-phrase matching is a whole-word check** using bash regex with word boundaries (`[[ "\$ISSUE_BODY" =~ (^|[^[:alnum:]_])\$TRIGGER_PHRASE([^[:alnum:]_]|$) ]]`). This prevents partial-word matches (e.g., `@infer` will not match `@inference`). The `/model` override is parsed with a bash regex (`/model[[:space:]]+([a-zA-Z0-9/_.:-]+)`).
- **Observability knobs** wired to the Infer CLI's viper config:
  - `inputs.debug` â†’ `INFER_LOGGING_DEBUG`. Flips the CLI's zap logger to DebugLevel, which gates a small set of stdout JSON-line events (`role: "user", hidden: true, kind: "system_reminder"` for reminder injections; `type: "compaction_started" | "compaction_completed"` for auto-compaction). The parser (`src/parser.ts`) admits the `compaction_*` lines and the runner surfaces them to the Actions log as a concise `[agent] context compaction startedâ€¦` line via a `Ticker.onMessage` listener â€” **independent of `mirror-agent-logs`**, so they show even with stdout muted (they flow on the otherwise-muted stdout, which is why a `debug`-on run previously logged nothing). Turn on when diagnosing why the agent didn't follow the prompt, whether reminders fired, or whether a run **hung inside compaction** (a `started` with no matching `completed`).
  - `inputs.compact-auto-at` â†’ `INFER_COMPACT_AUTO_AT`. Auto-compaction threshold as a percent of model context window (default 50, valid 20-100). Lower to compact earlier; raise to delay summarisation. (Lower thresholds compact more often, and each compaction is a full LLM round-trip that can itself stall â€” implicated in the issue #58 hang; the `recover` step is the safety net, a provider-call timeout in `inference-gateway/cli` would be the prevention.)
  - `inputs.dry-run` â†’ `INFER_DRY_RUN` (threaded into **both** node steps). Plan-only mode for local `act` testing, and the **only** mock-agent path (the former `use-mock-agent` input was removed). When true it sets `INFER_BIN` to the bundled mock agent (and `MOCK_MAKE_COMMIT` on the happy scenario), and the install/init/skills steps gate on `inputs.dry-run != 'true'`. The runner prints a "DRY RUN â€” the agent would be invoked with:" banner (system/task/reminder prompts + the resolved bash allow-list append), and `GithubClient` *simulates* every mutation (`[dry-run] would â€¦`, redactor still applied) while letting reads pass through (token optional; empty token â‡’ unauthenticated Octokit, reads fail-soft). The bash eyes/cooking steps branch on `$DRY_RUN`; the cooking step emits a synthetic `cooking_comment_id=999999999` so `updateZone`/`clearSpinner` still log (both short-circuit before any API call). Run via `task test:issue` / `test:comment` / `test:direct`, which drive `examples/local/*.yml` (`uses: ./`) through `act` â€” no `.env` needed.

```
â”śâ”€â”€ parser.ts       async generator over JSON-line streams

â”śâ”€â”€ runner.ts       run-agent entrypoint (spawn + ticker + cancel signal handler)
â”śâ”€â”€ recover.ts      recover entrypoint (always() work-salvage + finalize)

```

The three entrypoints (`runner.ts`, `recover.ts`, `post-results.ts`) each `ncc`-bundle to their own `dist/<name>/` and auto-run `main()` unless `VITEST` is set; `recovery.ts` is a **shared library** (no auto-run), imported by both `runner.ts` (for `sh`/`collectDiffStat`/`dumpAgentTail`/`setOutput`) and `recover.ts` (the full recovery flow) â€” keeping recovery out of `runner.ts` is what lets `recover.ts` import it without re-spawning the agent.

`tsconfig.json` is strict: `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `verbatimModuleSyntax`, `noPropertyAccessFromIndexSignature`. ESM only (`module: "nodenext"`), so all relative imports must use `.js` extensions even when importing `.ts` source (e.g. `from './ticker.js'`). `import type` is enforced via the `@typescript-eslint/consistent-type-imports` rule.

`dist/` is **built by ncc and committed to the repo** â€” consumers don't run `npm install`. CI verifies `git diff --exit-code dist/` after a fresh build. If you edit `src/`, run `npm run package` and commit the diff in the same PR.

`__tests__/` uses vitest. `__tests__/fixtures/mock-agent.mjs` is a standalone Node script that mimics the `infer agent` JSON-line stream for the chosen `MOCK_SCENARIO` â€” point `INFER_BIN` at it to drive the runner without the real CLI.

## Conventions

- **Commits must follow Conventional Commits** (`feat:`, `fix:`, `docs:`, `chore:` â€” see `.releaserc.yaml` for the release-impact mapping). `feat` â†’ minor, `fix`/`docs`/`chore` â†’ patch, breaking â†’ major.
- **The pinned CLI version** lives in `action.yml` at `inputs.version.default` (currently `v0.121.0`). Bump it via a `chore(deps)` commit.
- **Don't add new top-level files lightly** â€” this is a single-purpose action. New behavior almost always belongs inside `action.yml` (a new input + a step) rather than a new script or directory.
- **`.infer/` holds the maintainer-agent workspace.** The `.infer/*.yaml` files (`config.yaml`, `agents.yaml`, skills, shortcuts, â€¦) are **tracked/committed**; only runtime state (conversations, logs) is ignored, via the tracked `.infer/.gitignore`. Because the action's own `infer init`/`infer config set` steps rewrite files under `.infer/` at run time, the **"Configure Infer for headless run"** step hides those edits from git (`.git/info/exclude` for untracked consumer repos + `git update-index --skip-worktree` for tracked ones) so they never land in the agent's commits or trip the stopped-early dirty-tree check.
