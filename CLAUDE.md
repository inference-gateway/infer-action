# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This repo is a **GitHub composite Action** with a **TypeScript hot path**. The shipped surface is `action.yml` plus the pre-bundled ESM entrypoints under `dist/runner/`, `dist/salvage/`, and `dist/report/` (committed to git, built from `src/*.ts` with `bun build`). The "simple" steps (trigger detection, eyes reaction, cooking-comment post, CLI install, git config, cleanup) are still `shell: bash`. The three complex steps ÔÇö running the agent with a parallel ticker that mirrors `TodoWrite` to the issue comment, salvaging an interrupted run's unpushed work into a draft PR, and reporting the final result ÔÇö are `bun` invocations of the bundled scripts.

Consumers reference it as `inference-gateway/infer-action@<ref>` from their workflows. When a GitHub issue or comment containing the trigger phrase (`@infer` by default) fires, the action installs the `infer` CLI, sets up Bun, then runs `bun dist/runner/index.js` (which spawns `infer agent` and observes its JSON-line stream, debouncing TodoWrite updates to the cooking comment); on `if: cancelled() || failure()` it runs `bun dist/salvage/index.js` (which salvages an interrupted run's unpushed work into a draft PR ÔÇö surviving a job-timeout cancellation that the runner can't); and on `always()` it runs `bun dist/report/index.js` (which links the PR, computes the final status, and PATCHes a structured result footer ÔÇö including token usage and per-session cost ÔÇö into the same cooking comment).

## Common commands

The toolchain is **Bun, end to end** ÔÇö package manager, script runner, bundler, test runner, and the runtime the action ships. No npm/pnpm/yarn, and no Node in the build/test/runtime path; the only remaining Node is the manually-dispatched `Release` workflow (`semantic-release`, see the Bun note below). `task` wraps the most common combinations.

**Run everything inside the flox environment ÔÇö `bun`, `task`, and `node` are _not_ on the bare `PATH`.** The whole toolchain is provided by `.flox/env/manifest.toml` (Bun pinned to the same 1.3.13 as CI; `task` is `go-task`; `act`/`markdownlint-cli`/`jq`/`curl` and the `infer` CLI flake too ÔÇö there is no `node` at all). Either work inside an activated shell (`flox activate`, whose `on-activate` hook runs `bun install --frozen-lockfile` for you) or prefix one-off commands with `flox activate --` (e.g. `flox activate -- bun run test`, `flox activate -- task build`). A bare `bun ÔÇŽ`/`task ÔÇŽ` from a non-activated shell fails with "command not found". CI does **not** use flox ÔÇö it installs Bun via `oven-sh/setup-bun` at the same pin.

```sh
bun install --frozen-lockfile   # install from bun.lock
bun run package                 # bun build src/{runner,salvage,report}.ts -> dist/ (+ licenses)
bun run test                    # bun test
bun run typecheck               # tsc --noEmit
bun run lint                    # eslint .
bun run lint:md                 # markdownlint . (check-only; no --fix)
bun run format:check            # prettier --check .
bun run format:write            # prettier --write .
bun run all                     # format + lint + typecheck + test + package
```

`task` targets:

- `task build` ÔÇö `bun install --frozen-lockfile && bun run package`
- `task test:unit` ÔÇö `bun run test`
- `task test:mock SCENARIO=happy` ÔÇö runs the runner end-to-end against the mock agent (`__tests__/fixtures/mock-agent.mjs`). Useful for local iteration without a real `infer` CLI or GitHub token. Mock scenarios: `happy`, `failures`, `no-todos`, `empty`.
- `task test:issue` / `task test:comment` / `task test:direct` / `task test:all` ÔÇö `act`-based local tests that run the **working-tree** action (`uses: ./`) in `dry-run` mode against `examples/local/*.yml`. Require Docker + `act` only ÔÇö no `.env`/token (dry-run simulates all mutations; reads fail-soft). Pass a token with `-s GITHUB_TOKEN=$(gh auth token)` to resolve real reads. `task test:list` lists jobs without executing; `task setup` checks `act`/Docker and seeds `.env`.
- `task lint` ÔÇö `markdownlint --fix` (the auto-fixing local convenience; `bun run lint:md` is the same lint check-only, and is what CI runs. Both are separate from `bun run lint`, which is eslint over `src/`)
- `task clean` ÔÇö removes `/tmp/agent-output.txt`

Run a single test file: `bun test __tests__/failures.test.ts`
Run a single test by name: `bun test -t "drops envelope failures with an empty message"`

CI (`.github/workflows/ci.yml`) runs format-check Ôćĺ lint (eslint) Ôćĺ lint:md (markdownlint) Ôćĺ typecheck Ôćĺ test Ôćĺ package Ôćĺ `git diff --exit-code dist/` Ôćĺ mock smoke-test on every PR. The diff check fails if a contributor edited `src/` without rebuilding `dist/`.

**Bun is the runtime, not just the tooling ÔÇö no Node in the build, test, or runtime path.** Because this is a *composite* action (not a `using: node20` JavaScript action), the three TS steps are `shell: bash` steps that call `bun "$GITHUB_ACTION_PATH/dist/<name>/index.js"` directly ÔÇö sidestepping the GitHub-Actions "JS actions run on Node" limitation without the `configure-bun-action` binary-embedding hack. `action.yml` installs Bun on the consumer runner via `oven-sh/setup-bun` (pinned), and the runner step keeps `exec bun ÔÇŽ` so the bun process is the step's top-level process and still receives `SIGINT`/`SIGTERM` on a job-timeout cancel. The bundler is `bun build --target=bun` (see `scripts/bundle.mjs`), which also rewrites the shebang to bun and re-aggregates third-party `licenses.txt` (Bun has no native `--license` like ncc did). Tests run under `bun test` (`bun:test`'s Jest-compatible `mock`/`spyOn`; there is no `vi.stubEnv`, so `prompts.test.ts` saves/restores `process.env` by hand). The three entrypoints auto-run `main()` only when `import.meta.main` is true (bun sets it on the directly-executed entry, false on import), which is how `bun test` imports them without firing `main()`. CI installs Bun via `oven-sh/setup-bun`; locally Bun comes from the flox environment (`.flox/env/manifest.toml`). The committed `dist/` is byte-stable per Bun version, so CI and flox pin the same Bun (currently 1.3.13); a Bun bump regenerates `dist/` and must be committed. Two gotchas: (1) `bun run` does not run npm-style `pre<script>` hooks, so the `build:prompts` prerequisite (generating `src/prompts.gen.ts`) is chained explicitly inside `package`/`test`/`typecheck`/`lint`; (2) `bunfig.toml` sets `[run] bun = true` so `bun run` executes the node-shebanged dev CLIs (prettier/eslint/tsc/markdownlint) **under Bun** instead of shelling out to `node` ÔÇö without it the toolchain would still need Node on PATH. The lockfile is `bun.lock`. The one place Node remains is `.github/workflows/release.yml`, which runs `semantic-release` on `actions/setup-node` + `npm install -g` ÔÇö kept on Node deliberately (a heavyweight plugin tool whose manual, low-frequency release path isn't worth re-validating under Bun). **The timeout-recovery hot path (`SIGINT`/`SIGTERM` Ôćĺ cancel-marker Ôćĺ `child.kill('SIGKILL')` Ôćĺ `cancelled()`/`failure()` salvage + `always()` report, hardened by issues #58/#93/#94/#130) now runs under Bun ÔÇö treat it as the highest-risk surface when changing the runner, salvage, or report steps.**

Dry-run a build locally: set `dry-run: true` (and optionally `mock-agent-scenario: happy|failures|no-todos|empty`) on an action invocation. The action skips the CLI install/init/skills steps, points `INFER_BIN` at the bundled `__tests__/fixtures/mock-agent.mjs`, prints the resolved SYSTEM/TASK/REMINDER prompts, and **simulates** every GitHub mutation (`[dry-run] would ÔÇŽ`) while keeping reads real. Useful for eyeballing comment shape, the token-usage footer, and PR-link behavior of a build before cutting a release ÔÇö without burning provider tokens or mutating anything. (`dry-run` is the only mock-agent path; the old `use-mock-agent` input was removed in favor of it.)

Releases run via the manually-dispatched `Release` workflow (`.github/workflows/release.yml`), which invokes `semantic-release`. Versions are derived from conventional commit messages per `.releaserc.yaml`.

## Architecture: what `action.yml` actually does

The composite action is one linear pipeline. The bash steps are unchanged from earlier versions; the three `bun` steps (`run-agent`, `salvage`, `report`) are where the TypeScript hot path lives.

1. **`check-trigger`** ÔÇö scans `github.event.issue.{title,body}` or `github.event.comment.body` for the trigger phrase; skips if a bot authored the comment (recursion guard); extracts an optional `/model provider/name` override into `model_override`. Sets `triggered=true/false` ÔÇö every later step is gated on this.
2. **Eyes reaction** + **cooking comment** ÔÇö posts an "I'm cooking..." comment and captures its ID into `cooking_comment_id`. This is the single bot comment per run; everything downstream PATCHes it in place. Before posting, deletes any prior "I'm cooking" comments on the same issue (cleanup for runs that died mid-execution).
3. **Install Infer CLI** ÔÇö `curl | bash` from the `inference-gateway/cli` repo at the pinned `version` input, then **Initialize Infer userspace config** ÔÇö `infer init --userspace` seeds the CLI's *userspace* config layer in the runner's HOME (`~/.infer/config.yaml`), never the consumer's workspace, so the CLI always has a valid config base even in a repo with no committed `.infer/` (the run-agent step's `INFER_TOOLS_*` env vars still override it; the seed is idempotent ÔÇö skipped when `~/.infer/config.yaml` already exists). Then **Hide Infer workspace from git** ÔÇö adds `.infer/` to `.git/info/exclude` (untracked consumer repos) and `git update-index --skip-worktree`s any tracked `.infer/` files (this repo dogfooding itself), so the CLI's runtime writes under `.infer/` never dirty the working tree the agent commits from. The headless-run config ÔÇö auto-approving the `Write`/`Edit`/`Delete` tools (the CLI defaults them to require approval, which a headless run with no approval broker would otherwise block) and the `tools.web_fetch.allowed_domains` allow-list from the `web-fetch-domains` input ÔÇö is set via `INFER_TOOLS_*` env vars on the run-agent step (env vars take precedence over `.infer/config.yaml`), so the action no longer runs the *project* `infer init --overwrite` or `infer config set` (only the userspace seed above).
4. **Configure Git** as `github-actions[bot]`.
5. **Setup Bun** via `oven-sh/setup-bun` ÔÇö installs Bun on the runner for the run-agent/salvage/report bundles (there is no Node step; the runtime is Bun).
6. **`run-agent`** (TS, `dist/runner/index.js`) ÔÇö the core step. Invoked via `exec bun ÔÇŽ` so the **bun process is the step's top-level process** and actually receives `SIGINT`/`SIGTERM` when the job's `timeout-minutes` cancels it (a plain `bun ÔÇŽ` leaves bash as top-level, and bash swallows the signal). Spawns `infer agent -m $INFER_AGENT_MODEL "$TASK"` (or `$INFER_BIN` if set ÔÇö used by tests and by `dry-run: true`), tees stdout to `/tmp/agent-output.txt`, optionally mirrors it to the Actions log (off by default; `mirror-agent-logs`), and runs a **parallel ticker** over the JSON-line stream. The ticker dispatches to per-tool handlers ÔÇö `TodoWrite` renders the agent's todos as a markdown checklist and **throttles** PATCHes to the cooking comment's plan zone (ÔëĄ one per ~1.5 s, latest wins) and **persists the latest todos to `/tmp/infer-todos.json`** (the handoff the report step reads) ÔÇö plus, with `debug` on, a generic `Ticker.onMessage` listener that surfaces `compaction_started`/`compaction_completed` and reminder-injection events to the Actions log **regardless of stdout mirroring** (they flow on the muted stdout, so without this they stayed invisible even with debug on). On a normal exit the runner sets the `exit-code`/`run-duration-ms`/`result` outputs and returns. **It no longer recovers work or links PRs itself** ÔÇö salvage moved to the `salvage` step (7) and status/linking to the `report` step (8), so they survive a job-timeout cancellation. On `SIGINT`/`SIGTERM` (job timeout/cancel) the runner writes a **cancel marker** (`/tmp/infer-cancelled`), reaps the agent child, and exits **without** setting `exit-code` ÔÇö that marker (not the empty exit-code alone) is how the report step tells a genuine timeout from a runner crash or a skipped upstream step (the `salvage` step dumps the redacted transcript **breadcrumb**). The action still appends the agent's PR-workflow writes (`git add/commit/push/checkout/switch/fetch` + `gh pr create`/`gh pr ready`) onto the CLI's read-only baseline via `INFER_TOOLS_BASH_ALLOW_APPEND` (the agent opens its own PR early as a draft on the happy path), deliberately withholding `gh pr merge`/`close`/`edit`/`review`.
7. **`salvage`** (TS, `dist/salvage/index.js`, runs on `if: cancelled() || failure()`) ÔÇö the model-independent **work-salvage** safety net (`recoverUnpushedWork` in `src/recovery.ts`). It runs **only when the run did not finish gracefully** ÔÇö a clean agent exit (exit 0) skips it, so no draft PR is ever opened on a graceful finish (**issue #130**: the runner propagates the agent's exit code, so exit 0 Ôçĺ the step succeeds Ôçĺ neither `cancelled()` nor `failure()`). `cancelled()` covers a job-`timeout-minutes` stop; `failure()` a non-zero agent exit or a runner crash; GitHub runs both in the cancellation window (~4 min), so this survives a timeout that the **old in-`runner` recovery never did** (issue #58). If the agent left uncommitted/unpushed work and no PR exists, it commits, moves the work off `main` onto a `fix/issue-N` / `infer/auto-<run-id>` branch (or reuses the agent's branch / the PR head for `pull_request`), pushes, and opens a **draft** PR with a `buildPrBody`-synthesised body, exporting the URL as the `pr-url` output for the report step to link. It also dumps the redacted transcript **breadcrumb** (it only runs on an interrupted/failed run). **Fail-soft** (a salvage failure must not break the always() report step that follows); never merges; never pushes `main`/`master`; fork PRs are read-only and skipped. Its git/`gh` calls run with a 60 s timeout and `GIT_TERMINAL_PROMPT=0` so a hung push can't burn the cancellation budget. (A graceful exit-0 that left unpushed work is **not** salvaged here ÔÇö it surfaces as ÔÜá´ŞĆ stopped-early via the report step, deliberately retiring the old issue-#85 auto-PR-on-exit-0 net in favour of a human-in-the-loop signal.)
8. **`report`** (TS, `dist/report/index.js`, runs on `always()`) ÔÇö the single **finalize + report** step, merging what used to be the `recover` (finalize + PR-link) and `post-results` (footer) steps into one process: one env block, the status computed inline instead of handed across via step outputs. (1) **Status** (`finalizeStatus` in `src/recovery.ts`) from `run-agent`'s exit-code output + the runner's **cancel marker** (`/tmp/infer-cancelled`) ÔÇö emitted **first** so a later throw can never strand the action with empty status: the marker Ôçĺ a timeout Ôçĺ `timed-out=true`, `stopped-early=true`, exit-code **normalised to 0** (a salvaged timeout is a soft ÔÜá´ŞĆ, not a ÔŁî); an **empty** exit-code *without* the marker is a real ÔŁî (a crashed runner or a skipped/failed upstream step), never laundered into a soft timeout; the **stopped-early** signal (`detectStoppedEarly`) is otherwise any non-completed todo from the handoff file or ÔÇö with git ops on ÔÇö a still-dirty tree. (2) **Link the PR**: the draft the salvage step pushed (`INFER_SALVAGED_PR_URL`) if present, else the PR the agent opened on the happy path (`linkAgentPr`, with the thin-body backfill from `src/pr-body.ts`); `linkPr` writes the URL into the comment's middle zone + `pr-url` output and **returns** it for the footer. (3) **Footer**: failed tool calls from `/tmp/agent-output.txt` via a two-pass scan (pass 1 a `tool_call_id Ôćĺ tool_name` map from every `assistant.tool_calls[]`; pass 2 one row per failure, looked up via the map so envelope failures get a real tool name, **dropping any row whose error message is empty**), the **agent's final response** (`extractFinalResponse` in `src/response.ts`, redacted + capped at 16,000 chars, omitted when empty), **token usage** + **per-session cost** (`extractUsage` in `src/usage.ts`) ÔÇö written to `$GITHUB_STEP_SUMMARY` and PATCHed into the cooking comment's result zone (falling back to `POST` only if the PATCH fails or `cooking_comment_id` is empty). The status line is Ôťů Success, ÔŁî Failed (non-zero exit), or ÔÜá´ŞĆ Stopped early ÔÇö the last when `stopped-early` is set on an exit-0 run, or a **timed-out** run (always ÔÜá´ŞĆ, never ÔŁî, since exit-code was normalised to 0, with a "hit the job's time limit" note). ÔÜá´ŞĆ keeps the job's own conclusion unchanged. (4) Finally it ships **OTel** telemetry (best-effort). Fail-soft throughout: a context-read failure falls back to env-derived data and the status outputs are emitted before any linking/footer work.
9. **Cleanup** ÔÇö removes the temp files (incl. `/tmp/infer-todos.json`).

Key cross-cutting concepts to keep coherent when editing:

- **Section sentinels in the cooking comment body** ÔÇö `<!-- infer:plan-end -->` and `<!-- infer:result-start -->` split the body into three zones: plan (above plan-end), middle (between sentinels, used for the PR URL), result (below result-start, used for the final footer). Each writer only ever replaces its own zone via `GithubClient.updateZone()`: the `runner.ts` ticker owns the plan zone, and `report.ts` owns the middle (PR link) and result (footer) zones. This is what lets the runner and the report step update the same comment without stomping each other. See `splitZones` / `joinZones` in `src/github.ts`.
- **Ticker debounce semantics** are throttle-latest: first call sets a 1.5 s timer, subsequent calls update the pending value but don't reset the timer, the timer fires once with the latest value, and a new burst starts the next timer. Tuned so a tight loop of TodoWrites coalesces into ÔëĄ one PATCH per ~1.5 s while still surfacing the freshest state. See `throttleLatest` in `src/ticker.ts`.
- **The bash allow-list is "CLI baseline + a git-write append."** As of CLI v0.121.1 the CLI owns the read-only baseline (`tools.bash.mode.all.allow`: file reads, read-only git/gh, and `gh project list|view|item-list|field-list`), inherited by headless `standard` mode ÔÇö the action no longer ships its own read-only defaults. It only composes a small append in `src/bash-allow.ts` (`composeBashAllowAppend` / `GIT_WRITE_ALLOW`): the PR-workflow writes (`git add/commit/push/checkout/switch/fetch` + `gh pr create`/`gh pr ready`) when git ops are on, plus the `bash-allow-append` input, passed as the CLI's single append env var `INFER_TOOLS_BASH_ALLOW_APPEND`. `gh pr merge`/`close`/`edit`/`review` are never appended, so the agent can open and ready a draft PR but never merge it. Each entry is a Go regex anchored to the whole command (so `npm` matches only bare `npm`; write `npm( .*)?`); the CLI splits the list on `,` and `\n`. There is **no env-var path to replace the baseline** ÔÇö to do that, configure the CLI's `.infer/config.yaml` directly.
- **`enable-git-operations: false`** skips the salvage step entirely (its `if:` requires git ops) and omits PR-linking in the report step, *and* omits `GIT_WRITE_ALLOW` from the append, so the agent keeps only the CLI's read-only baseline (`bash-allow-append` still applies) and cannot push or open PRs by hand.
- **The runner owns the comment; the agent owns the branch and the PR on the happy path ÔÇö with the `salvage` step as a model-independent safety net for interrupted runs.** The system prompt does not ask the agent to comment on the issue (the runner mirrors TodoWrite and the report step posts the result), but it *does* tell the agent to commit on a `fix/issue-N` branch, run the repo's checks before each commit, and open the PR itself **early as a draft** with `gh pr create --draft --body-file` (writing the body to a file with the Write tool first, to avoid shell-quoting problems) and a real description, marking it ready with `gh pr ready` only when finished ÔÇö never merging it. Opening early means an interrupted run still leaves a **draft** PR with the pushed work (rather than nothing); the report step's read-only `stopped-early` check then surfaces a ÔÜá´ŞĆ status. **When a run is cancelled or fails**, the `salvage` step's `recoverUnpushedWork` (`src/recovery.ts`, see step 7) is the backstop: it commits/pushes whatever the agent left and opens the draft PR itself, never pushing `main` and never merging. On a **graceful exit-0**, salvage does **not** run, so a weak model that finishes without pushing is reported as ÔÜá´ŞĆ stopped-early rather than getting an auto-PR ÔÇö the old issue-#85 auto-PR-on-exit-0 net was deliberately retired to fix the issue-#130 spurious-PR-on-success bug. If the agent opened a PR with a thin body, the report step backfills one from the commit log (`src/pr-body.ts`) when it links it. `custom-instructions` is still appended verbatim.
- **The `infer` binary path is overridable via `INFER_BIN`** ÔÇö the integration test workflow and the local `task test:mock` target use this to substitute `__tests__/fixtures/mock-agent.mjs` for the real CLI.
- **Trigger-phrase matching is a whole-word check** using bash regex with word boundaries (`[[ "\$ISSUE_BODY" =~ (^|[^[:alnum:]_])\$TRIGGER_PHRASE([^[:alnum:]_]|$) ]]`). This prevents partial-word matches (e.g., `@infer` will not match `@inference`). The `/model` override is parsed with a bash regex (`/model[[:space:]]+([a-zA-Z0-9/_.:-]+)`).
- **Observability knobs** wired to the Infer CLI's viper config:
  - `inputs.debug` Ôćĺ `INFER_LOGGING_DEBUG`. Flips the CLI's zap logger to DebugLevel, which gates a small set of stdout JSON-line events (`role: "user", hidden: true, kind: "system_reminder"` for reminder injections; `type: "compaction_started" | "compaction_completed"` for auto-compaction). The parser (`src/parser.ts`) admits the `compaction_*` lines and the runner surfaces them to the Actions log as a concise `[agent] context compaction startedÔÇŽ` line via a `Ticker.onMessage` listener ÔÇö **independent of `mirror-agent-logs`**, so they show even with stdout muted (they flow on the otherwise-muted stdout, which is why a `debug`-on run previously logged nothing). Turn on when diagnosing why the agent didn't follow the prompt, whether reminders fired, or whether a run **hung inside compaction** (a `started` with no matching `completed`).
  - `inputs.compact-auto-at` Ôćĺ `INFER_COMPACT_AUTO_AT`. Auto-compaction threshold as a percent of model context window (default 50, valid 20-100). Lower to compact earlier; raise to delay summarisation. (Lower thresholds compact more often, and each compaction is a full LLM round-trip that can itself stall ÔÇö implicated in the issue #58 hang; the `salvage` step is the safety net, a provider-call timeout in `inference-gateway/cli` would be the prevention.)
  - `inputs.dry-run` Ôćĺ `INFER_DRY_RUN` (threaded into the run-agent, salvage, and report steps). Plan-only mode for local `act` testing, and the **only** mock-agent path (the former `use-mock-agent` input was removed). When true it sets `INFER_BIN` to the bundled mock agent (and `MOCK_MAKE_COMMIT` on the happy scenario), and the install/init/skills steps gate on `inputs.dry-run != 'true'`. The runner prints a "DRY RUN ÔÇö the agent would be invoked with:" banner (system/task/reminder prompts + the resolved bash allow-list append), and `GithubClient` *simulates* every mutation (`[dry-run] would ÔÇŽ`, redactor still applied) while letting reads pass through (token optional; empty token Ôçĺ unauthenticated Octokit, reads fail-soft). The bash eyes/cooking steps branch on `$DRY_RUN`; the cooking step emits a synthetic `cooking_comment_id=999999999` so `updateZone`/`clearSpinner` still log (both short-circuit before any API call). Run via `task test:issue` / `test:comment` / `test:direct`, which drive `examples/local/*.yml` (`uses: ./`) through `act` ÔÇö no `.env` needed.
  - **OpenTelemetry export** (`src/otel.ts`): zero-dep OTLP/HTTP JSON exporter that pushes per-run telemetry (token usage, cost, tool failures, run outcome, duration) to any OTLP-compatible collector. Disabled by default (empty endpoint). Controlled by `otel-exporter-otlp-endpoint` / `otel-exporter-otlp-headers` / `otel-service-name` / `otel-resource-attributes` / `otel-signals` / `otel-export-timeout-ms` inputs, which map to `OTEL_EXPORTER_OTLP_*` / `OTEL_SERVICE_NAME` / `OTEL_RESOURCE_ATTRIBUTES` / `OTEL_SIGNALS` / `OTEL_EXPORT_TIMEOUT_MS` env vars. Called from `report.ts` `main()` after the comment is posted. Best-effort, never fails the run. Honors dry-run. All strings pass through the redactor.

```
ÔöťÔöÇÔöÇ log-mirror.ts   planLogMirroring (stdout opt-in, stderr always)
ÔöťÔöÇÔöÇ bash-allow.ts   GIT_WRITE_ALLOW + composeBashAllowAppend (-> INFER_TOOLS_BASH_ALLOW_APPEND)
ÔöťÔöÇÔöÇ pr-body.ts      isThinPrBody + synthesize PR body from commit log (weak-model backstop)
ÔöťÔöÇÔöÇ recovery.ts     Shared lib: recoverUnpushedWork + linkPr/linkAgentPr + finalizeStatus + detectStoppedEarly + git/output helpers + dumpAgentTail
ÔöťÔöÇÔöÇ runner.ts       run-agent entrypoint (spawn + ticker + cancel signal handler)
ÔöťÔöÇÔöÇ salvage.ts      salvage entrypoint (cancelled()/failure() work-salvage -> draft PR)
ÔööÔöÇÔöÇ report.ts       report entrypoint (always() finalize + PR-link + footer + otel)
```

The three entrypoints (`runner.ts`, `salvage.ts`, `report.ts`) each bundle (via `bun build`, see `scripts/bundle.mjs`) to their own `dist/<name>/` and auto-run `main()` only when `import.meta.main` (run directly, not imported by a test); `recovery.ts` is a **shared library** (no auto-run), imported by `runner.ts` (for `sh`/`collectDiffStat`/`dumpAgentTail`/`setOutput`), `salvage.ts` (the `recoverUnpushedWork` salvage flow), and `report.ts` (`finalizeStatus`/`detectStoppedEarly` + `linkPr`/`linkAgentPr`) ÔÇö keeping recovery out of `runner.ts` is what lets `salvage.ts`/`report.ts` import it without re-spawning the agent.

`tsconfig.json` is strict: `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `verbatimModuleSyntax`, `noPropertyAccessFromIndexSignature`. ESM only (`module: "nodenext"`), so all relative imports must use `.js` extensions even when importing `.ts` source (e.g. `from './ticker.js'`). `import type` is enforced via the `@typescript-eslint/consistent-type-imports` rule.

`dist/` is **built by `bun build` and committed to the repo** ÔÇö consumers don't run an install step at all. CI verifies `git diff --exit-code dist/` after a fresh build. If you edit `src/`, run `bun run package` and commit the diff in the same PR.

`__tests__/` uses `bun test`. `__tests__/fixtures/mock-agent.mjs` is a standalone script (bun shebang) that mimics the `infer agent` JSON-line stream for the chosen `MOCK_SCENARIO` ÔÇö point `INFER_BIN` at it to drive the runner without the real CLI.

## Conventions

- **Commits must follow Conventional Commits** (`feat:`, `fix:`, `docs:`, `chore:` ÔÇö see `.releaserc.yaml` for the release-impact mapping). `feat` Ôćĺ minor, `fix`/`docs`/`chore` Ôćĺ patch, breaking Ôćĺ major.
- **The pinned CLI version** lives in `action.yml` at `inputs.version.default` (currently `v0.121.1`). Bump it via a `chore(deps)` commit.
- **Don't add new top-level files lightly** ÔÇö this is a single-purpose action. New behavior almost always belongs inside `action.yml` (a new input + a step) rather than a new script or directory.
- **`.infer/` holds the maintainer-agent workspace.** The `.infer/*.yaml` files (`config.yaml`, `agents.yaml`, skills, shortcuts, ÔÇŽ) are **tracked/committed**; only runtime state (conversations, logs) is ignored, via the tracked `.infer/.gitignore`. Because the Infer CLI writes workspace/runtime state under `.infer/` while the agent runs, the **"Hide Infer workspace from git"** step hides those writes from git (`.git/info/exclude` for untracked consumer repos + `git update-index --skip-worktree` for tracked ones) so they never land in the agent's commits or trip the stopped-early dirty-tree check. (The action no longer runs the *project* `infer init --overwrite`/`infer config set`; it seeds only the **userspace** `~/.infer/config.yaml` base via `infer init --userspace`, and configures tool approval and WebFetch domains via `INFER_TOOLS_*` env vars on the run-agent step, which take precedence over the config file.)
